如何在解析器中正确的处理编码问题

解析器（Parser）的工作通常是对一段文本进行结构化输出的操作。比如编程语言的语法解析器，抽象语法树（AST）、SQL、Mysql Protocol 等等。 对于文本解析器（SQL）而言，通常是根据Token获取相应的Message。既然是文本，自然就涉及到不同的文本编码，而Token一般是固定的标记字符，一般是ASCII 字符（当然也可以某种编码格式下的Unicode字符）。 那么，就可能在不同的编码环境中造成Token的解析失败。

比如 "0001101100010010" 这个字节序，在UTF16中，可以理解为&#6930，然后在ASCII环境中则可以理解为"00011011", "00010010" 这两个字符，这样的歧义就造成了解析器可能无法正常工作。

常用解决办法

  • 在协议中约定编码集，非约定的字符集不保证工作正确。虽然UTF16会造成ascii解析歧义，但是我们的Parser在进行工作的时候，对于ASCII字符同样使用UTF16编码则不会出现歧义。

  • 选择ASCII兼容的编码集，避免造成歧义。同样是Unicode，UTF8就是可以向下兼容ASCII编码的，因为UTF8中，每个字节的最高位都置为1（127+），而低于127的字符编码保持与ASCII一致，完成兼容处理。

  • 文本的本质是一段字节序列，如果应用必须要支持多种编码格式，那么可以采用预声明字符集的方法进行解析。比如在SQL的字符串定义中：
    _charset_name'strings'的方式预先指定一个字符集，在后续的解析中使用声明的字符集进行解析，从而避免歧义。这样，就可以在一段文本中使用多重字符编码。
